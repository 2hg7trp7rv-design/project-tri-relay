# TRI RELAY v0.4.1 新規プレイテスト手順

対象: ゲームを見たことがないT01〜T10
版: GitHubコミットSHAと不変配備URLを全記録へ固定する
目的: 最初の90秒で基本連鎖を理解し、本人の意思で次のランを始めるかを測る

## 記録の正本

Vercel本番URLの`/playtest`を観察者が開く。画面に不変配備へのリンクが出た場合は、セッション作成前にその`/playtest`へ移動する。T01〜T10と「正式ゲート」を選んで開始する。正式ゲートは、Vercelの`production`環境、40桁のGitHubコミットSHA、ビルドが記録した`*.vercel.app`配備URL、現在のoriginがすべて一致する場合だけ開始できる。ゲーム側でも開始時に同じ値を再照合し、不一致なら操作をロックする。正式セッション中の「実施手順」リンクも`main`ではなく、その記録のコミットSHAへ固定する。

観察記録は同じタブのSession Storageだけへ保存される。同意を「はい」と記録するまでセッションを作らない。T-ID、自由回答、観察者コード、端末名は送信せず、`/playtest`と`#playtest=…`のゲーム中はVercel Analytics、Speed Insights、カスタムイベント、クライアントエラー送信も停止する。各人の終了後、ブラウザを閉じる前にJSONを保存する。実データは`playtests/private/`などアクセス制限された場所へ置き、Gitへコミットしない。クリップボードコピーは回答原文をタブ外へ出すため、共有・端末同期先を確認してから使う。

コンソールから開始すると、テスト用プロフィールと途中保存を同じタブのSession Storageで新規作成する。通常プレイのLocal Storageは削除・上書きしない。途中保存には不透明なsession IDを結び付け、別タブや別T-IDのランを取り込まない。

`/playtest`の`noindex`はアクセス制御ではない。正式JSONはサーバー署名されておらず、集計器だけでは合成、事後編集、最初の10人からの選別を証明できない。運用時は信頼された観察者だけが不変配備へアクセスできるようVercel Deployment Protectionなどを設定し、テスト前にT01〜T10の固定名簿、順序、SHA、配備URL、保持期限を別のアクセス制限された追記不能記録へ確定する。集計器は欠損・重複・混在を拒否するが、この手続きの代わりではない。

## 禁止事項

- 観察者は、操作、色、周波数、共鳴、負荷、敵、強化を説明しない。
- 「そこを押して」「次は製造」などの誘導をしない。
- 失敗を防ぐために端末へ触らない。
- 再挑戦を促さない。
- 同僚や開発関係者を新規テスターに数えない。
- 不具合回を成功回として補正しない。原因と除外判断を別欄に残す。
- 結果画面では30秒間沈黙する。結果到達から単調時計で30.000秒以内に本人が結果画面の再挑戦を押した場合だけ、再挑戦候補として自動記録する。30.001秒以降は数えない。再挑戦しなかったセッションは結果記録から30秒経つまで確定できない。
- 途中離脱、クラッシュ、進行不能を成績から消さない。「中断ランを未完了で記録」を使う。

安全・アクセシビリティ上の補助が必要な場合は実施してよいが、補助内容を記録し、標準群とは分けて解釈する。

## 事前準備

各テスターは観察者コンソールから開始する。次を記録する。

- テスターID: T01〜T10
- 実施日時
- コミットSHA、配備URL
- 端末、OS、ブラウザ、画面寸法
- 音声ON/OFF、入力方法
- 観察者
- 既知の類似ゲーム経験（終了後に聞く）
- 同意、初見であること、開発関係者ではないこと
- 操作補助・アクセシビリティ補助の有無

開始前画面で、観察者コード、同意、初見、外部テスター、操作補助予定を必ず確定する。正式群は同意あり、初見、開発関係者ではない、操作補助なしだけを開始できる。ゲーム開始時の実viewportと言語で記録を更新する。正式群は幅320〜430、高さ568〜932の縦画面とし、10件のプレイ言語を混在させない。

## 観察手順

### 0〜5秒

ゲーム画面へ入った時点から説明せずに計時する。タイトル画面の開始ボタンを探す時間と、開始後に中央リレーを特定する時間を混同しない。

記録:

- 最初に見た場所
- 5秒以内に押す対象を特定したか
- 最初の入力時刻
- 間違って操作した対象

### 開始〜最初の撃破

沈黙したまま次を時刻付きで記録する。

- 採掘への最初の送電
- 製造への最初の送電
- 防衛への最初の送電
- 最初の有効弾薬生成
- 最初の撃破
- チュートリアル完了
- 迷い、連打、説明文の長時間読解、誤タップ

「最初に見た場所」「迷い・連打・長時間読解」は専用欄へ記録し、なかった場合も空欄にせず「なし」とする。誤って操作した対象も、なければ「なし」とする。

自動イベント`first_kill`は撃破のゲーム内時刻を記録する。観察者は壁時計での時刻も別に記録する。

### 90秒

`active_90s_reached`は、ページが前面かつゲームが明示ポーズされていない能動時間の累計90秒で発火する。ガイド中に考えている時間、インターミッション、強化選択中を含む。タブ非表示、明示ポーズ、バックグラウンド時間は含めない。

観察者の壁時計90秒は`壁時計90秒地点`へ別記し、能動時間と混同しない。壁時計の各欄は、発生した場合は開始からの秒数、ラン終了まで発生しなかった場合は`未到達`を選ぶ。空欄は「未到達」ではなく観察漏れとして`INCOMPLETE`になる。自動証拠と発生／未発生が矛盾する壁時計記録も`INCOMPLETE`になる。

自動記録の`wallSeconds`は単調時計による「ページがロードされていた区間」の経過であり、リロード間の空白を復元しない。正式な実時間判断は観察者の壁時計欄と、sessionの`startedAt / completedAt`を優先し、自動`wallSeconds`を真の壁時計と呼ばない。

### ラン終了

勝敗に関係なく、結果画面を閉じる前に次を記録する。

- 能動時間、壁時計時間
- 最初の撃破までの時間
- 回転数、回転/秒
- 生産的パルス率
- 過負荷回数、オーバードライブ回数
- 最終ビルド
- 表示された事実ベースの振り返り

結果画面で沈黙を続け、本人が自発的に再挑戦した場合だけ「任意2ラン目」とする。観察者が質問または促しをした後の再挑戦は別集計にする。再挑戦ボタンを押した記録は必ず、その同時刻に始まる次ランの記録と対にする。次ランを取得できなければクラッシュ／進行不能として残し、成功群には入れない。最初のラン後に開始画面から別の`start`ランを作ることは認めず、記録可能なランは完全な連鎖の最大6本までとする。

結果画面到達前に本人が終了を希望した場合は、観察者コンソールへ戻り「中断ランを未完了で記録」を実行する。初撃破なしは`null`であり、欠損値ではない。

クラッシュや保存領域の故障でcheckpoint自体が復元不能な場合は、クラッシュ／進行不能件数を1以上にして「復元不能ランを技術事故として閉じる」を使う。ラン記録がなくても技術事故を消さず、そのセッションは`NO-GO`へ残す。件数0のままこの経路は使えない。自動検知または観察者が一度保存した手順逸脱は追記専用で、同じセッション内では削除・上書きできない。誤入力なら参加者へ提示する前にセッション自体を作り直す。

設備・敵のシルエット確認では、最初に無彩色・記号番号だけのカードを提示する。回答中は正解名をDOMへ表示しない。「回答を確定して採点欄を表示」を押してカードを閉じ、端末を観察者へ戻してから正解名付きの採点欄を使う。

## 終了後の質問

質問はこの順序で、一度だけ行う。選択肢を先に読まない。

1. 「このゲームで何をしていましたか？」
2. 「採掘、製造、防衛はどうつながっていますか？」
3. 「中央の装置を回した理由は何ですか？」
4. 「2ウェーブ目から増えたルールは何でしたか？」
5. 「危険な状態を画面のどこで判断しましたか？」
6. 「もう一度、自分から遊びたいですか？ そう思う／思わない理由は何ですか？」

回答は要約せず、本人の語句を短く記録する。沈黙や不明も空欄にせず「（無回答）」など事実を明示する。ただし回答内容を送信せず、テスト記録だけで管理する。

## 採点規則

- 基本連鎖の合格: 「採掘で鉱石」「鉱石を製造で弾薬へ」「弾薬を防衛で使う」の3関係を、誘導なしで本人の言葉に含む。単に3語を順番に読むだけでは不合格。
- W2共鳴の合格: 次の周波数／色と送電先を合わせると出力が増えることを本人が説明する。色が変わったとだけ述べる場合は不合格。W2未到達も不合格で、分母から外さない。
- 5秒リレーの合格: 開始後5秒以内に、唯一の操作対象である中央リレーを指す、または正しく操作する。
- 重要情報は、脅威、都市、現在経路、次経路、鉱石、弾薬を別々に採点する。設備は採掘・製造・防衛を別々に採点する。ラン記録後に`/playtest`が表示する無彩色の設備A〜Cと敵1〜4だけを同じ条件で提示し、ラッシャー、サッパー、ジャマー、ウォーデンを別々に採点する。色や欄の名称を先に見せない。集約ゲートは各組の全項目が「はい」の人だけを1人として数える。最初の5秒にまだ登場していない敵4種を見たことにしない。
- 不明、無回答、観察者の言い換えが必要だった回答は「未記録」ではなく不合格として確定する。真に記録自体がない場合だけ「未記録」とする。

## 合格条件

最初の10人で、すべてを満たした時だけ次段階へ進む。

- 8/10以上が、誘導なしで採掘→製造→防衛を説明できる。
- 8/10以上が、最初の撃破を25秒以内に達成する。
- 7/10以上が、能動時間90秒へ到達する。
- 5/10以上が、促されずに2ラン目を開始する。
- 8/10以上が、W2で追加された周波数／共鳴の意味を概ね説明できる。
- 8/10以上が、開始5秒以内に中央リレーを特定する。
- 8/10以上が、別記した重要情報確認と終了後の敵シルエット確認を通過する。
- 重要操作を妨げる文字切れ、重なり、誤タップが320×568または390×844で再現しない。
- クラッシュ、進行不能、復帰破損、バックグラウンド進行が0件。

一項目でも未達ならNO-GOとし、最も早い離脱点を一つ直して、新しい未経験者10人で再試験する。同じ人の学習後成績を新規理解率へ混ぜない。

T01〜T10を都合よく入れ替えない。プロトコル逸脱、補助、同意未確認、開発関係者、コミット混在があれば集計器は`INCOMPLETE`とする。成績不足、クラッシュ、UI妨害は`NO-GO`であり、代替テスターを足して消さない。

10件保存後、同一SHA・同一配備のJSONを次で集計する。

```bash
npm run playtest:summary -- playtests/private/T01.json playtests/private/T02.json playtests/private/T03.json playtests/private/T04.json playtests/private/T05.json playtests/private/T06.json playtests/private/T07.json playtests/private/T08.json playtests/private/T09.json playtests/private/T10.json
```

出力は`INCOMPLETE / NO-GO / GO`、件数、閾値、欠損・整合性エラー、共通コミットSHA、共通配備URL、入力データの正規化SHA-256を表示するが、回答原文は表示しない。壊れたJSONの構文エラーも原文をstderrへ表示しない。JSON Schemaは交換形式の構造を示し、時系列や自動証拠との対応を含む最終判定は同じ版の集計器がより厳格に検証する。テストfixtureは集計ロジックの検証専用であり、実際の役員会判断には事前固定した名簿と信頼された観察者の実JSONだけを使う。ローカルJSON自体には暗号学的な真正性保証がない。

## 次段階

10人ゲート通過後、30〜50人で次を確認する。

- 任意再挑戦率
- 平均セッション時間と中央値
- 1分到達率
- 3ビルドの選択分布と勝率
- 敗北後の再挑戦
- 端末・画面寸法別の離脱

この段階を通るまで、12ウェーブ化、敵6種化、強化18種化、開始プロトコル、広告、CrazyGames Basic Launch申請へ進まない。
